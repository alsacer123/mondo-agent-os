# GitHub 任务说明

这个仓库的 Issue 和任务不围绕“再加一个功能”展开，而是围绕一个客户结果展开：

> 用户能不能把混乱工作说出来，并让 AI 持续接住这个工作现场？

因此，提交任务时请优先描述真实使用过程、用户感受和工作流断点，而不是先描述技术实现。

## 当前优先收集什么

### 真实工作场景接入

使用 Issue 模板：`真实工作场景接入`

适合提交：

- 你现在有哪些项目总是需要反复给 AI 解释。
- 你希望 AI 接住哪类日常工作。
- 你现在的 Daily、待办、项目文档或客户资料如何组织。
- 你不希望 AI 自动碰哪些边界。

好的任务描述应该包含：

- 用户角色：内容生产者、职场工作者、自由职业者、创业者等。
- 当前工作现场：长期方向、当前项目、卡点、下一步。
- 期望变化：用了 Mondo 后，用户应该少做什么、多看见什么。
- 风险边界：哪些动作必须先确认，哪些信息不能写入。

## 当前不优先收集什么

这些不是永远不做，只是不属于公开底座的当前优先级：

- 新建完整聊天 UI。
- 自建模型网关或 API 聚合器。
- 自建复杂定时任务平台。
- 直接开源完整商业角色包。
- 把 Mondo 做成替代 Cherry Studio、Codex、Claude Code、Cursor 或 Cline 的主体工具。

如果相关问题来自真实使用阻塞，可以记录为现象；但任务描述仍应回到用户结果，例如“用户无法在 Cherry Studio 关闭后完成早间整理”，而不是直接写“做一个后台 Agent 平台”。

## 如何判断一个任务是否该做

优先做同时满足下面三点的任务：

- 它能让普通用户更容易完成初始化建模。
- 它能让 Agent 更稳定地读写 Daily、行动池、项目状态和下一步。
- 它能减少用户第二天重新解释背景的次数。

暂缓做下面这类任务：

- 主要证明技术能力，但用户体验没有明显变化。
- 需要大幅增加系统主体复杂度。
- 只适配单个用户的特殊偏好，暂时看不出可复用角色包价值。
- 会让用户误以为 Mondo 是全自动员工，而不是可审计的工作现场底座。


## 当前阶段判断口径

当前阶段不以功能数量作为成功标准。

当前阶段只看：

- 用户能否完成一次真实初始化。
- 用户是否理解生成出来的工作现场。
- 用户是否愿意把第二天的工作继续交给这套系统承接。
- 我们能否从多个真实场景中抽象出可复用角色包。

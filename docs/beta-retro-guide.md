# 第一轮内测复盘指南

这份指南用于内测结束后 30 分钟内复盘。目标不是总结得漂亮，而是决定下一轮到底该改哪里。

复盘只围绕一个问题：

> 用户有没有感到 AI 开始接住他的工作现场？

## 复盘输入

先准备这些材料：

- `docs/beta-feedback-form.md` 的填写结果。
- 用户原话，尤其是困惑、惊喜、抗拒和不信任的表达。
- 生成出来的工作区结构。
- 安装和配置耗时。
- 主持人现场观察。

不要只看“功能是否跑通”。用户感受比功能数量更重要。

## 第一步：判定结果

把本次内测归到一个结果：

- 通过：用户感受到 AI 开始接住工作现场。
- 部分通过：能完成初始化，但日常流转不清晰。
- 不通过：用户只把它理解成任务记录工具。
- 不通过：安装或配置阶段阻塞。

如果无法判断，按“不通过”处理。第一轮宁愿严格一点。

## 第二步：定位问题类型

### 1. 配置问题

表现：

- 用户卡在 Cherry Studio / MCP / Python / 路径配置。
- 还没进入真实口述，耐心已经被消耗。
- 用户开始问“我是不是要懂技术”。

下一轮优先改：

- `docs/cherry-studio-mcp.md`
- `docs/user-first-day-guide.md`
- README 的第一天入口
- 安装前检查和错误提示

不要立刻做完整聊天 UI，除非 3 次以上内测都卡在同一入口。

### 2. 引导问题

表现：

- 用户不知道该怎么讲长期方向。
- 用户只说零散待办，讲不出项目现场。
- 问题太像填表，用户不愿意继续说。
- 用户说完后觉得“这不像我”。

下一轮优先改：

- onboarding 提问顺序。
- `docs/beta-host-runbook.md`
- `docs/user-first-day-guide.md`
- 助手开场提示词。

判断标准：修改后，用户是否更容易把脑子里的混乱倒出来。

### 3. 底座问题

表现：

- 系统把项目事实、下一步、等待事项混在一起。
- 所有输入都被塞进今日任务。
- Daily、行动池、项目下一步之间流转不清。
- 用户第二天需要重新解释背景。

下一轮优先改：

- onboarding 写入规则。
- `mondo_route_input` 和写回工具边界。
- Daily / 行动池 / 项目状态协议。
- `scripts/verify_beta_flow.py` 的验证覆盖。

判断标准：修改后，Agent 是否更稳定地恢复上下文和写回下一步。

### 4. 角色包问题

表现：

- 底座能跑，但用户的工作类型需要专门语言。
- 内容生产、职场推进、客户交付等场景的“下一步”判断明显不同。
- 用户能理解系统，但觉得“还没贴住我的工作方式”。

下一轮优先改：

- 角色包问题清单。
- 场景提问模板。
- 私有陪跑 SOP。

不要急着把完整角色包开源。先记录共性，至少 3 个相似用户后再抽象。

## 第三步：写出下一轮只改一件事

每次复盘最后只写一个最高优先级动作：

```text
下一轮只改：

为什么是它：

用户证据：

完成后如何验证：
```

如果同时想改 5 件事，说明还没有真正复盘。

## 第四步：决定写到哪里

- 用户原话和内测结果：写入 `docs/beta-feedback-form.md` 或 GitHub Issue `第一轮内测反馈`。
- 可复用任务：按 `docs/github-task-guide.md` 写成 Issue。
- 路线判断：更新 `docs/execution-roadmap.md`。
- 工具缺陷：补验证脚本或 MCP / CLI 行为。
- 场景共性：先放私有角色包沉淀，不急着开源。

## 不要做的事

第一轮内测结束后，不要马上：

- 因为一个用户卡住就重做整个架构。
- 因为用户提到自动化就做复杂定时平台。
- 因为配置不顺就直接做完整聊天客户端。
- 因为一个场景很特殊就开源角色包。
- 用技术指标掩盖用户没有感受到价值。

## 复盘通过标准

一次复盘完成，必须得到：

- 本次内测结果判定。
- 最高优先级问题类型。
- 一个下一轮只改的动作。
- 对应的用户证据。
- 验证这个改动是否有效的方法。

没有这些，就不要进入下一轮开发。

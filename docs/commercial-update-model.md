# 商业化与更新机制

## 基本判断

这个项目最适合采用：

> 免费开源骨架 + 付费陪跑接入 + 长期维护订阅

免费仓库负责建立信任、解释方法论、展示基础结构。真正收费的是把客户自己的真实业务接进去，并让系统持续进化。

## 产品分层

### 免费层：GitHub 公开仓库

内容：

- AI Personal Work OS 基础结构
- 五文件项目现场
- Daily / 行动池 / 项目写回规则
- 安全边界
- 初始化脚本
- 示例项目

目标：

- 获客
- 建立专业信任
- 让用户理解“AI 缺的不是 Prompt，而是项目现场”
- 筛选有真实需求的人

### 付费层 1：一次性接入陪跑

内容：

- 诊断客户真实工作场景
- 设计目录结构和项目入口
- 改写客户专属 `AGENTS.md`
- 建立项目规则、状态、下一步和决策日志
- 跑通一个真实项目

交付物：

- 客户本地 AI 工作系统
- 1-3 个真实项目接入
- 客户专属运行规则
- 第一次 Daily / 项目推进流程
- 验收清单

### 付费层 2：年度维护订阅

内容：

- 定期发送系统升级说明
- 新增模板、规则、诊断表、Agent 流程
- 客户本地 Agent 根据更新说明自检和升级
- 定期复盘客户系统运行情况

目标：

- 降低服务多人客户的边际成本
- 让客户系统持续进化
- 把一次性交付变成长期关系

## 更新通道设计

### 方案 A：邮件更新

流程：

```text
服务方发布更新邮件
  -> 客户邮箱接收
  -> 客户本地 Agent 定期读取指定邮箱/转存文件
  -> Agent 解析更新说明
  -> 对照客户本地系统执行升级建议
  -> 生成变更记录和待确认项
```

邮件内容结构：

```text
标题：[AI Work OS Update] 2026-06-03 Daily 回流规则升级

适用对象：
- 使用行动池和 Daily 的客户

更新内容：
- 新增 Daily 晚间回流检查
- 增加“项目动作回项目，跨项目动作回行动池”的判断

Agent 执行建议：
1. 读取工作区 _索引.md
2. 检查 40_Daily/_行动池.md 是否有回流规则
3. 检查 Daily 模板是否包含晚间回流
4. 如缺失，提出补丁，不要直接覆盖用户已有内容

需要人工确认：
- 是否启用每晚自动复盘
```

优点：

- 客户容易理解
- 不要求复杂部署
- 适合早期服务

风险：

- 邮箱读取权限和安全边界要谨慎
- 不能让 Agent 盲目自动改客户系统

### 方案 B：私有 GitHub 仓库

流程：

```text
服务方维护私有更新仓库
  -> 客户 Agent 定期 pull 或读取 release notes
  -> 对照本地系统生成升级建议
  -> 客户确认后应用
```

适合：

- 高客单价客户
- 企业客户
- 需要版本管理和变更追踪的客户

优点：

- 版本清晰
- 可追踪变更
- 能区分免费版和付费版

风险：

- 客户需要 GitHub 访问能力
- 对非技术用户门槛稍高

### 方案 C：更新包文件

流程：

```text
服务方发送 update-pack.md / zip
  -> 客户放入指定 inbox
  -> Agent 读取 update-pack
  -> 生成升级建议
  -> 人确认后执行
```

适合：

- 非技术个人用户
- 不想配置邮箱/GitHub 的客户

## 推荐早期方案

第一阶段不要做复杂自动化。

建议先用：

1. 免费 GitHub 公开仓库获客。
2. 付费客户建立专属本地工作系统。
3. 每月邮件发送一次“系统升级说明”。
4. 客户把邮件内容转存到本地 `00_Inbox/updates/`。
5. Agent 读取更新包，生成升级建议。
6. 人确认后再改文件。

这样既能体现长期服务价值，又不会一开始陷入邮箱授权、自动同步、安全权限等复杂问题。

## 不要过早自动化的部分

- 自动读取客户邮箱。
- 自动改客户项目规则。
- 自动覆盖客户 Daily 或行动池。
- 自动处理客户隐私数据。
- 自动部署远程服务。

这些可以作为后期高级服务，不是第一阶段必须能力。

## 对外表达

可以这样说：

> 免费仓库给你一套 AI 工作系统骨架。  
> 陪跑服务帮你把真实业务接进去。  
> 年度维护会持续给你发送系统升级包，让你的本地 Agent 根据新规则自检和进化。

核心卖点不是“我给你一堆模板”，而是：

> 你的 AI 工作系统不是一次性搭完，而是会随着我的方法论更新持续升级。

# 第一位内测候选触达清单

这份清单只解决一件事：在真正发出邀请前，帮助主持人从身边候选人里选出最值得先测的一个人。

不要把它做成销售漏斗，也不要同时约很多人。第一轮只需要找到一个能暴露真实问题的用户。

## 1. 先列 3 个候选人

每个候选人只写一句话，不要先做长分析。

```text
候选人 A：
- 角色：
- 手里可能正在推进的 2 个项目：
- 为什么可能适合：
- 最大风险：

候选人 B：
- 角色：
- 手里可能正在推进的 2 个项目：
- 为什么可能适合：
- 最大风险：

候选人 C：
- 角色：
- 手里可能正在推进的 2 个项目：
- 为什么可能适合：
- 最大风险：
```

优先选“项目混乱但愿意说真话”的人，不优先选“最懂 AI”的人。

## 2. 先发一句轻触达

不要一开始解释 MCP、Agent OS、技术架构。先确认对方是否有真实工作现场和 20-30 分钟时间。

```text
我最近在测一个个人 AI 工作系统，不是待办工具，主要想验证一件事：
能不能让 AI 先听你讲清楚长期方向和手上的项目，然后帮你生成一个以后能持续接手的工作现场。

我想先找一个真实用户跑 20-30 分钟。你手里如果现在有至少 2 个正在推进的项目，而且愿意直接口述真实情况，我想邀请你做第一轮内测。
```

如果对方问“我需要准备什么”，再发 `docs/beta-invitation-message.md` 的完整版本。

## 3. 看回复，不要急着约时间

对方回复后，先按下面三类判断。

### 可以进入准入记录

用户回复里出现这些信号：

- 有至少 2 个真实项目；
- 说得出现在卡在哪里；
- 愿意口述，而不是要求你先给模板；
- 能接受空工作区测试；
- 愿意第二天再看一次系统有没有接上。

下一步：运行或打开候选用户准入记录。

```bash
mondo-os beta-intake
```

或使用：

- `docs/beta-user-intake-template.md`

### 先补问一句

用户有兴趣，但信息还不够清楚时，只补问一个问题：

```text
你现在手里最真实、最需要推进的 2 个项目分别是什么？不用整理，直接说名字和现在卡在哪一步就行。
```

如果对方仍然只讲泛泛需求，暂时不要进入第一位内测。

### 暂时不适合第一位内测

这些回复先放下：

- “我就想要一个待办清单/提醒工具”；
- “我没有具体项目，你先演示一下”；
- “我不想让系统写本地文件”；
- “我主要想评测技术架构”；
- “我现在先不想花 20-30 分钟讲具体事情”。

这不是否定用户价值，只是不适合作为第一轮验证样本。

## 4. 选定后再正式邀请

通过或部分通过后，再发：

- `docs/beta-invitation-message.md`
- `docs/user-first-day-guide.md`

然后在正式内测前跑：

```bash
python scripts/verify_all.py
```

## 5. 本轮只记录一个判断

第一位用户触达结束后，只产出一个判断：

```text
第一位候选用户：
准入结论：通过 / 部分通过 / 不通过
最大风险：
下一步：约内测时间 / 补问一个问题 / 暂时不进入第一轮
```

不要因为一个候选人不合适就改产品。先确认是样本问题，还是系统入口问题。

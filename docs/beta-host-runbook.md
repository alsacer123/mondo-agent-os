# 第一轮内测主持人 Runbook

这份文档给主持人使用。目标是稳定跑完第一轮内测，不在现场临时发挥。

## 内测目标

验证普通用户能不能在不理解技术架构的情况下，把混乱工作口述成一个 AI 能接手的工作现场。

不是验证：

- 高级自动化。
- 完整角色包。
- 定时任务。
- 多 Agent 调度。
- 复杂知识库检索。

## 开始前 5 分钟

确认：

- 用户使用的入口：Cherry Studio / Codex / Claude Code / 其他。
- 工作区路径：建议空文件夹，例如 `D:/MondoWorkOS`。
- 用户至少有 2 个真实项目。
- 已运行本地 dry-run：

```bash
python scripts/verify_beta_flow.py
```

主持人提醒用户：

```text
你不用整理成漂亮文字，直接把真实情况说出来就行。
这次不是测你会不会用工具，而是测系统能不能接住你的工作。
```

## 第 1 段：启动建模

让用户说：

```text
请开始帮我建立个人 AI 工作系统，工作区放在 D:/MondoWorkOS。
```

观察：

- 助手是否先问长期方向和项目现场。
- 助手是否避免解释 MCP/API/命令行。
- 用户是否知道接下来该说什么。

## 第 2 段：长期方向采集

引导用户说：

- 未来 3-6 个月最重要的方向。
- 当前最混乱、最想让 AI 接住的工作。
- AI 可以帮忙的事。
- AI 不能自动做的事。

主持人不要替用户总结太多。让用户用自己的语言讲。

记录：

- 用户是否能讲清长期方向。
- 助手是否能复述用户的重点。
- 助手是否识别边界。

## 第 3 段：项目现场采集

引导用户说：

- 正在推进的项目。
- 等待中的项目。
- 暂停但仍重要的项目。
- 每个项目现在做到哪一步。
- 每个项目下一步是什么。
- 哪些是本周主线。

观察：

- 助手是否把项目事实和下一步分开。
- 助手是否区分主线、保活、等待、暂停。
- 助手是否避免把所有东西塞进今日任务。

## 第 4 段：确认初始化草稿

让用户检查：

- 长期方向像不像自己。
- 项目有没有漏掉。
- 项目有没有分错。
- 本周主线是否合理。
- 第一个下一步是否能直接执行。

如果用户说“不像我”，不要强行写入。记录失败点。

## 第 5 段：写入工作区

用户确认后再写入。

检查是否生成：

- `_Identity/长期方向.md`
- `_Identity/定位.md`
- `_Identity/内容原则.md`
- `40_Daily/_行动池.md`
- 当天 Daily
- 项目 `_项目规则.md`
- 项目 `_总览.md`
- 项目 `_当前状态.md`
- 项目 `_next.md`
- 项目 `决策日志.md`

## 第 6 段：用户复述

让用户用自己的话解释：

- Daily 是什么。
- 行动池是什么。
- 项目下一步是什么。

如果用户只能说“这就是待办”，说明产品表达还没打中。

## 第 7 段：收尾提问

问用户：

```text
刚才哪一刻你觉得它真的有用？
哪一步最困惑？
你愿不愿意明天继续用它做早间整理？
```

## 记录方式

使用：

- `docs/beta-feedback-form.md`
- 或 GitHub Issue 模板：`第一轮内测反馈`

## 通过标准

通过：

- 用户完成初始化建模。
- 用户看懂初始化草稿。
- 用户愿意确认写入。
- 用户能大致解释 Daily / 行动池 / 项目下一步。
- 用户愿意第二天继续用。

不通过：

- 用户只把它理解成待办工具。
- 用户无法讲出当前项目。
- 助手把所有输入都变成今日任务。
- 配置阶段阻塞，无法进入建模。

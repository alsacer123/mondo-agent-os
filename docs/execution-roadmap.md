# 执行路线图

这份路线图只围绕一个长期目标推进：

> 帮客户把混乱工作变成一个 AI 能持续接手的工作现场。

Mondo Agent OS 不是独立聊天客户端，也不是完整 Agent Runtime。它要成为 Agent 生态里的工作系统底座：让 Cherry Studio、Codex、Claude Code、Cursor、Cline 或其他 Agent 能读懂用户的工作现场，并把状态、下一步、Daily 和决策持续写回。

## 当前定位

开源仓库只做底座：

- 工作现场协议。
- 初始化建模流程。
- Daily / 行动池 / 项目状态 / 下一步流转规则。
- Agent 使用说明、CLI / MCP / Skill 适配。
- 安全边界、诊断和验证脚本。

完整商业场景包暂不放入公开底座。内容生产、工作推进、客户交付等角色包先作为内部方法沉淀，等真实使用场景足够稳定后再决定开放边界。

## 阶段 0：公开底座可运行

目标：让一个干净工作区能被初始化、检查、导出上下文，并能被 Codex、Claude Code、Cherry Studio 或其他具备本地文件能力的 Agent 读取和写回。

已完成：

- 最小 Markdown OS 结构。
- CLI 初始化、诊断、扫描、输入路由、实时状态和 Agent context。
- 可选 MCP 工具入口。
- Agent 驱动的初始化建模说明。
- runtime、MCP、onboarding flow 验证脚本。

通过标准：

```bash
python scripts/verify_runtime.py
python scripts/verify_mcp.py
python scripts/verify_onboarding_flow.py
```

## 阶段 1：真实工作初始化可用

目标：验证普通用户能不能在不理解 MCP、API、CLI 或文件结构的情况下，把长期方向和当前项目录入成可流转的工作现场。

执行顺序：

1. 用户在 Cherry Studio 或其他 Agent 客户端里请求建立个人 AI 工作系统。
2. Agent 先引导用户讲长期方向，再讲所有当前项目，而不是只输入一条待办。
3. Agent 收集身份、项目、卡点、边界和本周主线。
4. 用户确认后，Agent 按 Mondo OS 规则写入 `_Identity/`、行动池、当天 Daily 和真实项目文件。
5. 第二天用户可以从早间整理继续，不需要重新解释全部背景。

成功标准：

- 用户能说清“我现在有哪些项目、做到哪一步、最卡哪里”。
- 系统能把项目事实、下一步、等待事项、Daily 临时事项分开。
- 用户能看懂生成后的工作现场。
- 用户第二天能从早间整理继续，而不是重新解释背景。

## 阶段 2：按真实反馈修底座

目标：只修真实使用暴露的问题，不为了技术完整感扩张系统。

优先修：

- 安装和配置是否卡住普通用户。
- Cherry Studio Agent 使用说明是否足够清楚。
- 初始化提问是否能让用户自然讲出项目现场。
- 写入后的文件是否让用户看得懂。
- Daily、行动池、项目下一步的区别是否被用户理解。
- Agent 是否会误把临时事项写成项目事实。

暂不优先：

- 自建聊天 UI。
- 自建模型网关。
- 自建完整 Agent Runtime。
- 自建复杂定时平台。
- 把商业场景包全部开源。

## 阶段 3：从多种真实场景抽象角色包

目标：从不同真实用户里找共性，而不是围绕单个用户无限定制。

建议覆盖：

- 内容生产型个人品牌。
- 职场工作推进型。
- 自由职业或客户交付型。
- 创业者多项目推进型。

每轮记录：

- 用户最自然的口述方式。
- 最容易混乱的项目类型。
- 哪些下一步写法真的能推动执行。
- 哪些文件用户会看，哪些文件只是给 Agent 读。
- 哪些场景需要角色包，哪些只需要底座规则。

阶段产物：

- 更新初始化问题。
- 抽象 2-3 个高频角色包。
- 明确哪些能力适合开源，哪些应保留为陪跑服务或私有模块。

## 阶段 4：沉淀角色包

目标：把“1000 个人有 1000 个场景”拆成少数可复用工作类型。

角色包不是固定模板，而是 Agent 的工作协议起点。每个角色包至少定义：

- 用户常见工作现场。
- 初始化时必须问的问题。
- 项目状态应该怎么写。
- 下一步怎样才算可执行。
- Daily 应该从哪里抓任务。
- 哪些事项必须进决策日志。
- 哪些动作需要用户确认。

第一批可探索方向：

- 内容生产型个人品牌。
- 职场工作推进型。
- 自由职业客户交付型。
- 创业者多项目推进型。

## 阶段 5：适配 Agent 生态

目标：让用户带着自己已有的 Agent 来用 Mondo，而不是被迫换工具。

优先适配：

- Cherry Studio：普通用户图形界面入口，使用模型、记忆和本地文件能力直接操作 Mondo 工作区；MCP 作为可选辅助。
- Codex / Claude Code：技术用户和重执行场景，通过 AGENTS / Skill / CLI 使用。
- Cursor / Cline：代码和项目文件场景，通过项目规则和上下文文件使用。

可选适配：

- Windows Task Scheduler / n8n：做早间整理、晚间收口等外部触发。
- Obsidian 插件或轻量 Dashboard：读取 `.mondo/live-state.json` 展示实时状态。

原则：

- Agent 负责推理、对话和执行。
- Mondo 负责工作现场、写回规则和状态连续性。
- 适配器可以增加，底座不要变成膨胀主体。

## 阶段 6：判断是否成熟

只有当下面这些标准稳定成立，才算接近成熟：

- 一个普通用户能在主持人帮助下完成初始化。
- 第二天用户能从系统给出的早间整理继续工作。
- 白天新增事项能被正确归属。
- 晚间收口能更新项目状态和明天第一步。
- Agent 换工具后仍能读懂同一套工作现场。
- 项目文件不会泄露密钥、密码、服务器信息或无关资产。
- 至少 3 个角色包经过真实使用验证。

## 当前唯一下一关

让公开底座稳定支持一次真实工作初始化：

> 用户能不能从“我只是和 AI 聊天”，跨到“AI 开始接住我的工作现场”？

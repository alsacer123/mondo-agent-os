# 客户体验与长期目标

Mondo Agent OS 的长期目标不是让用户多学一个工具，而是让用户感受到：

> 我原来只会和 AI 聊天，现在 AI 开始真的接住我的工作了。

## 目标客户结果

客户最终应该获得一个可持续流转的个人工作现场：

- 长期方向被写清楚。
- 当前项目被完整录入。
- 每个项目有状态、规则和下一步。
- Daily 只承接今天要做的动作。
- 行动池承接跨天动作、等待、将来也许和灵感。
- 重大判断进入决策日志。
- Agent 每次进入时知道先读什么、能做什么、不能做什么、做完写回哪里。

## 第一天体验

用户打开 Cherry Studio 或其他 Agent 客户端，说：

```text
请开始帮我建立个人 AI 工作系统。
```

系统不要求用户理解 MCP、API、CLI 或文件结构，而是先引导用户讲清：

- 长期想把工作和生活变成什么样。
- 未来 3-6 个月最重要的方向。
- 当前手上所有正在推进、等待中、暂停但重要的项目。
- 每个项目现在做到哪一步。
- 每个项目当前卡点和下一步。
- 哪些是本周主线，哪些只是保活。
- AI 可以做什么，不能自动做什么。

用户感受到的不是填表，而是把脑子里的混乱倒出来。

## 初始化完成后的体验

系统生成工作现场后，用户应该能看到：

- 我的长期方向。
- 我的项目列表。
- 每个项目的当前状态。
- 每个项目的唯一下一步。
- 今天要做什么。
- 哪些事情放行动池。
- 哪些事情只是等待。

用户感受到的是：

> 我终于能看清楚自己手上的事了。

## 每日流转体验

早上：

```text
帮我做今天早间整理。
```

系统读取行动池、昨天 Daily、今天 Daily 和项目下一步，生成今天可以执行的任务池。

白天：

```text
我刚和客户聊完，他下周要看方案。
```

系统判断这是项目状态、下一步动作、等待事项还是 Daily 临时事项，并说明写入建议。

晚上：

```text
帮我晚间收口。
```

系统引导用户确认：

- 今天完成了什么。
- 未完成项回流哪里。
- 哪些项目状态变了。
- 哪些重大判断要进决策日志。
- 明天第一步是什么。

## 不做什么

Mondo 不做新的聊天客户端，不替代 Cherry Studio、Codex、Claude Code、Cursor、Cline 或其他 Agent。

Mondo 只提供：

- 工作现场协议。
- 初始化建模流程。
- Daily / 行动池 / 项目状态 / 下一步的流转规则。
- MCP 和 CLI 工具。
- 给不同 Agent 读取和写回的边界。

谁的 Agent 强，就让谁来执行。Mondo 负责让执行不乱、状态不断、工作能接着走。

## 结果验证

公开底座只验证一件事：

> 普通用户能不能在不理解技术架构的情况下，把自己的长期方向和当前项目录入成一个 AI 能接手的工作现场。

成功标准：

- 用户能完成一轮初始化口述。
- 系统能生成 `_Identity/`、行动池、当天 Daily 和初始化项目文件。
- 用户能理解 Daily、行动池、项目下一步三者的区别。
- 用户第二天能从早间整理继续，而不是重新解释背景。

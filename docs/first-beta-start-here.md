# 第一位内测用户启动包

这页只服务一件事：带第一位真实用户跑完一次初始化建模。

不要在第一次内测里验证高级自动化、复杂定时任务、完整角色包或多 Agent 调度。第一位用户只需要感受到：

> 我把混乱工作说出来以后，AI 开始接住我的工作现场。

## 使用顺序

### 1. 发出邀请

邀请前先判断用户是否适合第一轮内测：

- `docs/beta-user-selection.md`
- `docs/beta-user-intake-template.md`

第一位用户要有真实项目现场，并且愿意把混乱工作口述出来。不要为了测试顺利而选择一个没有真实工作压力的人。

先用：

- `docs/beta-invitation-message.md`

确认用户愿意参与 20-30 分钟，并准备：

- 一个可以聊天的 AI 客户端，优先 Cherry Studio。
- 一个空工作区，例如 `D:/MondoWorkOS`。
- 至少 2 个真实项目。
- 一段不需要整理成模板的真实口述。

### 2. 内测前自检

主持人先在仓库根目录运行：

```bash
python scripts/verify_beta_flow.py
```

这只验证本地 onboarding 流程能跑通。它不能替代真实用户内测。

完整配置自检见：

- `docs/beta-config-preflight.md`

如果 GitHub Issue 创建、浏览器授权或网络状态不稳定，先生成一份本地执行包：

```bash
python scripts/prepare_first_beta_pack.py
```

安装后也可以运行：

```bash
mondo-os beta-pack
```

生成的 `.mondo/first-beta-run-pack.md` 可以当场作为主持人执行清单。第一位用户内测不应该被 GitHub Issue 提交动作阻断，Issue 可以事后补建。

### 3. 现场开场

主持人按：

- `docs/beta-host-runbook.md`

开场时提醒用户：

```text
你不用整理成漂亮文字，直接把真实情况说出来就行。
这次不是测你会不会用工具，而是测系统能不能接住你的工作。
```

然后让用户从这句话开始：

```text
请开始帮我建立个人 AI 工作系统，工作区放在 D:/MondoWorkOS。
```

### 4. 只观察关键客户结果

第一轮不要追求功能完整，只观察：

- 用户是否能自然讲出长期方向。
- 用户是否能讲出至少 2 个真实项目。
- 助手是否把项目事实和下一步分开。
- 助手是否避免把所有输入变成今日任务。
- 用户是否看懂初始化草稿。
- 用户是否愿意确认写入工作区。
- 用户是否愿意第二天继续用早间整理。

完整检查项见：

- `docs/beta-test-checklist.md`

### 5. 记录反馈

内测结束后立即记录：

- `docs/beta-session-notes-template.md`
- `docs/beta-feedback-form.md`
- `docs/beta-retro-guide.md`

如果用户愿意公开反馈，也可以用 GitHub Issue 模板：

- `第一轮内测反馈`

记录时优先保留用户原话，尤其是：

- 第一次觉得系统有用的瞬间。
- 最困惑的一步。
- 是否把它理解成“工作现场”，还是只理解成“待办工具”。

## 通过标准

第一位用户通过，不代表产品成熟。只代表这条路值得继续。

通过条件：

- 用户完成初始化建模。
- 用户看懂生成的工作现场。
- 用户能大致解释 Daily、行动池、项目下一步。
- 用户愿意第二天继续用。

失败也有价值。下面这些失败应该直接进入下一轮迭代：

- 配置阶段阻塞，无法进入建模。
- 用户不知道该怎么口述。
- 助手把长期方向、项目事实、下一步混在一起。
- 用户只觉得这是一个任务记录工具。
- 用户不信任系统写入工作区。

## 内测后下一步

内测结束后，不要立刻扩展功能。

先回答三个问题：

1. 用户有没有感到“AI 开始接住我的工作现场”？
2. 哪个环节让用户最困惑？
3. 这个问题是底座问题、引导问题，还是角色包问题？

再决定是修改 onboarding、修改文档、修改 MCP 工具，还是沉淀角色包。
